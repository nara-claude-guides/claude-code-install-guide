## 1️⃣ 기본 원칙: 명확성 > 간결성

### 🤔 왜 짧은 게 항상 좋은 게 아닐까요?

Claude는 **마음을 읽지 못해요.** 짧고 모호한 요청은 짧고 모호한 답을 낳아요.

### 🎬 비교 실습

**❌ 모호한 프롬프트:**

```
파일 정리 코드 짜줘
```

**✅ 명확한 프롬프트:**

```
현재 폴더에 있는 .jpg 파일들을 촬영 날짜(EXIF) 기준으로
"YYYY-MM" 폴더로 자동 이동시키는 Python 스크립트를 만들어줘.
- 라이브러리: Pillow
- 날짜 정보 없으면 "unknown" 폴더로
- 진행 상황을 print로 출력
```

### 🧱 명확성을 만드는 5가지 요소

| 요소 | 질문 | 예시 |
|------|------|------|
| 🎯 **목적** | 무엇을 하려는가? | "사진 정리 자동화" |
| 📥 **입력** | 무엇이 주어지는가? | ".jpg 파일 100장" |
| 📤 **출력** | 무엇이 결과인가? | "YYYY-MM 폴더 구조" |
| 🛠 **제약** | 무엇을 써야 하는가? | "Pillow, Python 3.10+" |
| 🚫 **예외** | 어떤 경우 처리는? | "EXIF 없으면 unknown" |

### 💡 한 가지 비밀

> **"잘 모르겠는 부분은 너가 정해줘"** 라고 명시하면, Claude가 합리적인 기본값을 선택해서 진행해요.

```
사진 정리 스크립트 만들어줘. 라이브러리 선택은 너가 정해줘.
```

### 🏋️ 실습 미션

다음 모호한 요청을 명확하게 다시 써보세요:

| 모호 ❌ | 명확 ✅ (직접 작성) |
|--------|-------------------|
| "TODO 앱 만들어줘" | ___ |
| "이거 빨라지게 해줘" | ___ |
| "테스트 코드 짜줘" | ___ |

## 2️⃣ 역할 부여(Role Prompting) & Few-shot

### 🎭 Role Prompting이 뭐예요?

Claude에게 **"~인 척 해줘"** 하고 페르소나를 부여하는 기법이에요.

```
너는 10년차 Python 백엔드 시니어 개발자야.
방금 신입 개발자가 짠 코드를 코드리뷰해줘.
초보자도 이해할 수 있게, 친절하지만 핵심은 짚어서.
```

### 비교: 역할 있음 vs 없음

| 같은 코드 리뷰 요청 | 결과 |
|-------------------|------|
| 🤷 역할 없음 | 일반적인 칭찬·지적 섞임 |
| 👨‍💻 시니어 백엔드 개발자 역할 | 성능·보안·확장성 관점 강조 |
| 👩‍🏫 신입 멘토 역할 | 친절한 설명 + 학습 자료 추천 |

### 🎬 실습: 같은 코드, 다른 페르소나

먼저 분석할 코드 만들기:

```
review_target.py 파일을 만들어줘.
사용자 비밀번호를 받아서 해시 없이 그대로 파일에 저장하는 함수가 들어있어.
```

이제 3가지 역할로 리뷰 요청:

```
1. 너는 보안 컨설턴트야. review_target.py를 검토해줘.
2. 너는 시니어 백엔드 개발자야. 같은 파일을 리뷰해줘.
3. 너는 친절한 코딩 멘토야. 신입 개발자가 이해할 수 있게 같은 파일을 리뷰해줘.
```

→ **세 답변을 비교**해보세요. 어조, 강조점, 추천 자료가 모두 달라요!

### 📚 Few-shot: "이런 식으로 해줘" 예시 주기

말로 설명하기 어려운 스타일은 **예시 1~2개로** 보여주는 게 빨라요.

**❌ 말로 설명:**

```
함수 주석을 한국어로 써줘.
근데 너무 길지 않게, 핵심만, 명령형으로.
```

**✅ 예시 보여주기 (Few-shot):**

```
다음 스타일로 함수에 docstring을 달아줘:

예시 1)
def add(a, b):
    """두 수의 합을 반환한다."""
    return a + b

예시 2)
def fetch_user(user_id: int) -> User:
    """user_id로 사용자를 조회한다. 없으면 None."""
    ...

이제 src/utils.py 의 함수들에 같은 스타일로 추가해줘.
```

### 🎯 Few-shot이 강력한 순간

| 상황 | 왜 효과적? |
|------|-----------|
| 🎨 일관된 스타일 적용 | 패턴을 직접 보여주는 게 빠름 |
| 🌐 특정 출력 포맷 강제 | JSON 키 이름·구조 통일 |
| 🗣️ 말투·어조 학습 | 회사 문서 톤 따라하기 |
| 🧪 엣지 케이스 처리 | "이런 입력엔 이렇게" 명시 |

> 💡 **황금 비율**: 예시는 **2~3개**가 베스트. 너무 많으면 컨텍스트 낭비, 1개면 일반화 안 됨.

## 3️⃣ Chain of Thought & Plan Mode 활용

### 🧠 Chain of Thought (CoT)란?

**"답만 말하지 말고, 풀이 과정도 보여줘"** 라고 요청하는 기법이에요.

### ❌ 결론만:

```
이 SQL 쿼리 최적화해줘.
```

→ Claude가 즉답으로 코드만 던짐. 왜 그렇게 했는지 모름.

### ✅ 단계별 사고:

```
이 SQL 쿼리를 최적화하려고 해. 다음 순서로 답해줘:
1. 현재 쿼리에서 느릴 만한 부분 진단
2. 가능한 최적화 옵션 3가지 비교 (장단점)
3. 최종 추천안과 그 이유
4. 개선된 쿼리 코드
```

→ 풀이 과정이 투명해져요. **틀린 가정**도 쉽게 발견 가능!

### 🪄 Plan Mode: CoT의 시스템 버전

Claude Code의 **Plan 모드**는 CoT를 **시스템 차원**에서 강제해요.

| 일반 모드 | Plan 모드 |
|----------|-----------|
| 답하면서 즉시 파일 수정 | 계획서 작성 → 사용자 승인 → 실행 |
| 빠르지만 위험 | 느리지만 안전 |
| 시연·강의에 부적합 | 시연·복잡한 작업에 최적 |

### 🎬 Plan 모드 켜기

```
Shift + Tab     # 모드 순환 (Default → Plan → AcceptEdits)
```

또는 시작 시:

```bash
claude --permission-mode plan
```

### 실습: Plan 모드로 복잡한 작업 시켜보기

```
간단한 가위바위보 게임을 Python으로 만들어줘.
- 사용자 vs 컴퓨터
- 3판 2선승
- 점수 기록
- 종료 시 통계 출력
```

Plan 모드라면:

1. 📝 Claude가 마크다운 **계획서** 작성
2. 👀 학습자가 단계 검토
3. 💬 부족한 부분에 코멘트
4. ✅ 승인 후 실행

> 🎓 **강의 활용**: 학습자가 "AI가 어떻게 사고하는지" 단계별로 볼 수 있어요!

## 4️⃣ 출력 형식 지정 & 검증 요청 실습

### 📐 출력 형식을 명시하면 일관성이 올라가요

**❌ 형식 미지정:**

```
이 프로젝트의 의존성 정리해줘.
```

→ 어떤 답은 표, 어떤 답은 줄바꿈, 어떤 답은 JSON... 매번 달라요.

**✅ 형식 지정:**

```
이 프로젝트의 의존성을 다음 마크다운 표로 정리해줘:

| 패키지 | 버전 | 용도 | 필수/선택 |
|--------|------|------|----------|
```

### 🎯 자주 쓰는 출력 형식 지정

| 상황 | 프롬프트 예시 |
|------|--------------|
| 📊 비교 자료 | "마크다운 표로" |
| 📋 체크리스트 | "체크박스 형태로 (`- [ ]`)" |
| 🤖 자동화 입력 | "JSON 객체로, 키는 영문 snake_case" |
| 📝 문서 초안 | "한국어 존댓말, 단락당 3문장 이내" |
| 🔢 단계별 안내 | "Step 1, Step 2 형식으로 번호 매겨" |

### 🧪 검증 요청: "스스로 점검해줘"

코드 작성 후 **Claude에게 자기 검증**을 시키면 품질이 확 올라가요.

```
방금 작성한 함수에 대해 스스로 다음을 점검해줘:

1. 🐛 엣지 케이스 빠진 거 없나? (빈 입력, None, 음수 등)
2. ⚠️ 보안 이슈 (SQL Injection, XSS, 권한 누락)
3. 🎨 네이밍이 일관적인가?
4. 📝 docstring/주석이 충분한가?
5. ✅ 실제로 어떤 입력을 넣어보면 검증되는지 예시 3개

문제 발견되면 코드 수정도 같이.
```

### 🔁 반복 검증 사이클

```
1차: 코드 작성 (요청)
   ↓
2차: 자기 검증 (코드 + 점검 결과)
   ↓
3차: 사용자가 추가 케이스 제시
   ↓
4차: 재검증 + 수정
```

> 💡 **비유**: 글쓰기 = 초고 → 퇴고. 코드도 똑같아요!

### 🏋️ 실습 미션

이전에 만든 `calculator.py`(또는 아무 코드)에 대해 자기 검증을 요청해보세요:

```
calculator.py를 시니어 개발자 시점으로 자기 검증해줘.
체크리스트는 다음 표 형식으로:

| 항목 | 통과/미흡 | 개선안 |
|------|-----------|--------|
```

## 5️⃣ CLAUDE.md 소개 및 직접 작성 실습

### 🤔 CLAUDE.md가 뭐예요?

**프로젝트의 안내문**이에요. Claude가 매 세션 시작할 때 자동으로 읽어요.

### 비유로 이해하기

| 회사 입사 | Claude Code |
|----------|-------------|
| 사규집 / 입사 안내문 | `CLAUDE.md` |
| 팀 위키 | 글로벌 `~/.claude/CLAUDE.md` |
| 부서별 매뉴얼 | 하위 폴더 `CLAUDE.md` |

### 📍 CLAUDE.md 위치별 우선순위

```
1. ~/.claude/CLAUDE.md          ← 모든 프로젝트 공통 (글로벌)
2. ./CLAUDE.md                  ← 이 프로젝트 전용 (팀 공유)
3. ./src/components/CLAUDE.md   ← 특정 영역 전용
```

### ✨ 자동 생성으로 시작하기

```
/init
```

→ Claude가 프로젝트를 분석해서 `CLAUDE.md` 초안을 만들어줘요.

### 🛠 CLAUDE.md 좋은 예시

```markdown
# Project: My TODO App

## 🎯 프로젝트 개요
- React + TypeScript + Tailwind 기반 할일 관리 앱
- 백엔드: Firebase (Firestore + Auth)

## 📦 기술 스택
- React 18, TypeScript 5
- Vite, Tailwind CSS
- Firebase 10

## 📐 코딩 컨벤션
- 함수형 컴포넌트 + Hooks 사용
- 변수: camelCase, 컴포넌트: PascalCase
- 모든 export는 default 대신 named export
- 한국어 주석 OK, 변수/함수명은 영문

## 🚫 하지 말아야 할 것
- class 컴포넌트 작성 금지
- console.log 커밋 금지 (eslint로 차단)
- any 타입 사용 금지

## 🧪 테스트
- 테스트 프레임워크: Vitest
- 모든 PR은 npm test 통과 필수
- 커버리지 70% 이상 유지

## 🎨 디자인
- Figma: [링크]
- 색상 팔레트는 tailwind.config.js 참조
```

### 🏋️ 실습 미션: 직접 CLAUDE.md 작성

```
지금 폴더에 CLAUDE.md를 만들어줘.
- 프로젝트: 가위바위보 Python 게임 (방금 만든 것)
- 코딩 스타일: PEP 8, 한국어 주석
- 테스트: pytest 필수
- 절대 하지 말 것: print 디버깅 남기기
```

작성 후 점검:

```
!cat CLAUDE.md
```

### 🎯 좋은 CLAUDE.md의 특징

| ✅ 좋은 예 | ❌ 나쁜 예 |
|----------|----------|
| 구체적 ("Vitest 사용") | 모호 ("테스트 잘 짜기") |
| 금지사항 명시 | 허용만 나열 |
| 짧게 (~200줄 이내) | 너무 길면 컨텍스트 낭비 |
| 자주 업데이트 | 한 번 쓰고 방치 |

> ⚠️ **CLAUDE.md는 컨텍스트를 차지해요.** 너무 길면 토큰 비용이 올라가니 **핵심만**!

## 6️⃣ 슬래시 커맨드 핵심 명령어 실습

### 🎯 자주 쓰는 TOP 5

| 순위 | 명령어 | 한 줄 설명 |
|------|--------|-----------|
| 1 | `/help` | 사용 가능한 모든 명령어 보기 |
| 2 | `/clear` | 대화 히스토리 완전 초기화 |
| 3 | `/compact` | 히스토리 압축 (요약하여 컨텍스트 확보) |
| 4 | `/status` | 현재 환경·인증·모델 한눈에 |
| 5 | `/init` | CLAUDE.md 자동 생성 |

### 🎬 실습: 5개 명령어 차례로 실행

#### Step 1. /help — 명령어 메뉴 열기

```
/help
```

→ 현재 세션에서 사용 가능한 **모든** 슬래시 명령어가 표시돼요.

> 💡 **핵심 팁**: 명령어 외울 필요 없어요. **`/help`만 기억**하세요!

#### Step 2. /status — 환경 점검

```
/status
```

→ 현재 폴더, 모델, 권한 모드, MCP 서버 상태 전체 표시.

> 🩺 자세한 읽는 법은 [/status 가이드](./status-reading-guide.md) 참고!

#### Step 3. /clear vs /compact — 차이 체감

먼저 대화를 길게 쌓아보세요:

```
> Python의 리스트와 튜플 차이를 자세히 설명해줘
> 딕셔너리와 셋도 비교해줘
> 클래스와 인스턴스 차이도 알려줘
```

이제 두 명령을 비교해보세요:

| 명령어 | 동작 | 언제? |
|--------|------|-------|
| `/clear` | **모든 대화 삭제** (백지) | 새 주제로 완전히 넘어갈 때 |
| `/compact` | **요약하여 압축** (핵심 보존) | 같은 작업 계속하는데 컨텍스트 부족 |

```
/compact
```

→ 상태바의 컨텍스트 % 가 확 줄어드는 게 보여요!

#### Step 4. /init — CLAUDE.md 자동 생성

```
/init
```

→ Claude가 폴더를 스캔해서 `CLAUDE.md` 초안을 만들어줘요. **5분 만에 완성!**

#### Step 5. 보너스 명령어

```
/model        # 모델 변경 (Opus / Sonnet / Haiku)
/permissions  # 권한 모드 변경
/cost         # 현재 세션 비용 확인
/usage        # 플랜 사용량 확인
/exit         # 세션 종료
```

### 🎯 종합 시나리오: 새 작업 시작 루틴

```
1. claude                       (셸에서 시작)
2. /status                      (환경 확인)
3. /init                        (CLAUDE.md 없으면 생성)

5. > 첫 작업 요청...
6. (컨텍스트 80% 도달 시)
7. /compact                     (압축)
8. > 다음 작업...
9. /exit                        (종료)
```

## 🆘 자주 막히는 지점

### Q1. 같은 프롬프트인데 결과가 매번 달라요

**A.** 정상이에요! AI는 **확률적**으로 답해요. 같은 결과가 필요하면:

```
방금 답변과 동일한 형식·구조로 다시 만들어줘.
```

또는 결과를 파일로 저장 후 재사용.

### Q2. CLAUDE.md를 썼는데 Claude가 무시하는 것 같아요

**A.** 점검 포인트:

```
/status            # Memory files 항목에 CLAUDE.md가 보이는지
!cat CLAUDE.md     # 내용이 정상인지
```

위치가 **Working Directory 루트**가 맞는지 확인하세요.

### Q3. Plan 모드인데 그냥 코드를 바로 실행해요

**A.** 권한 모드 체크:

```
/permissions       # 현재 모드 확인
Shift + Tab        # 순환해서 Plan으로
```

### Q4. /compact 했더니 중요한 정보가 사라졌어요

**A.** 다음에는 보존할 내용을 명시하세요:

```
/compact 데이터베이스 스키마와 API 엔드포인트 목록은 반드시 보존
```

### Q5. Few-shot 예시를 넣었는데도 다른 스타일로 답해요

**A.** 예시 개수를 늘려보세요. **2~3개**가 황금률입니다. 그리고 명시적으로:

```
위 예시 1, 2번과 정확히 같은 형식으로 작성해줘.
다른 스타일은 절대 사용하지 마.
```

### Q6. 프롬프트가 너무 길어지면 효과가 줄어드나요?

**A.** 원칙은 **"필요한 만큼만"** 입니다. 컨텍스트가 길어지면 핵심이 묻혀요. 반복적인 컨텍스트는 **CLAUDE.md로 옮기고**, 프롬프트엔 **그 작업에만 필요한 정보**를 담으세요.
